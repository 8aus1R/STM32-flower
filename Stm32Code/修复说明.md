# 修复说明

## 说明

本文档用于记录当前 `STM32` 自动浇花系统工程中已经确认的代码问题、建议修改方式，以及阈值控制模块的预期行为。

本次修复目标是：

- 消除确定性的编译风险
- 修复明确的逻辑错误
- 保持原有功能和工程结构不被大幅改动
- 统一阈值设置页面的显示与实际调节逻辑

## 需要修改的文件

本次需要修改以下 4 个文件：

1. `User/mqtt/mqtt.c`
2. `User/oled/bsp_oled_iic.c`
3. `SYS/my_usart/usart3.c`
4. `SYS/menu/menu.c`

## 详细修复内容

### 1. `User/mqtt/mqtt.c`

#### 问题 1：`memset` 参数顺序错误

原代码中存在类似写法：

```c
memset(clientid, 128, 0);
memset(username, 64, 0);
memset(passward, 64, 0);
memset(ServerIP, 32, 0);
```

这类写法的第三个参数为 `0`，实际不会清空缓冲区。

建议改为：

```c
memset(clientid, 0, sizeof(clientid));
memset(username, 0, sizeof(username));
memset(passward, 0, sizeof(passward));
memset(ServerIP, 0, sizeof(ServerIP));
```

#### 问题 2：阈值自增自减逻辑错误

原代码如果存在下面这类写法：

```c
Threshold.TempMax = Threshold.TempMax++;
Threshold.TempMax = Threshold.TempMax--;
```

这类代码不会按预期修改变量值。

建议改为直接增减，并增加边界判断，例如：

```c
if (Threshold.TempMax < 99) Threshold.TempMax++;
if (Threshold.TempMax > 0) Threshold.TempMax--;
```

同样方式处理以下阈值字段：

- `Threshold.TempMax`
- `Threshold.LightMin`
- `Threshold.SoliMin`
- `Threshold.HumiMin`
- `Threshold.SGP30Max`

建议范围：

- 温度阈值：`0 ~ 99`
- 光照阈值：`0 ~ 100`
- 土壤湿度阈值：`0 ~ 100`
- 空气湿度阈值：`0 ~ 100`
- 气体阈值：`0 ~ 1000`

#### 问题 3：缓冲区清空长度错误

原代码如果存在：

```c
memset(mqtt_CMDOutPtr + 2, 0x00, sizeof(mqtt_CMDOutPtr + 2));
```

则这里取到的是指针大小，不是剩余缓冲区长度。

建议改为实际剩余长度，例如：

```c
memset(mqtt_CMDOutPtr + 2, 0x00, 398);
```

具体长度仍应以实际数组定义为准。

### 2. `User/oled/bsp_oled_iic.c`

#### 问题：浮点显示缓冲区过小

原代码如果存在：

```c
uint8_t Data[] = " ";
sprintf((char *)Data, "%.3f", Fnum);
```

则 `Data` 空间明显不足，存在越界风险。

建议改为：

```c
uint8_t Data[16] = {0};
sprintf((char *)Data, "%.3f", Fnum);
```

如果后续需要显示更长字符串，也可以适当扩展缓冲区长度。

### 3. `SYS/my_usart/usart3.c`

#### 问题 1：阈值加减逻辑错误

和 `mqtt.c` 一样，如果存在：

```c
x = x++;
x = x--;
```

都需要改成普通的加减，并带上上下限保护。

例如：

```c
if (Threshold.LightMin < 100) Threshold.LightMin++;
if (Threshold.LightMin > 0) Threshold.LightMin--;
```

#### 问题 2：访问了不存在的结构体成员

如果代码中存在：

```c
Threshold.SmogMax
```

则这会导致编译失败，因为当前 `THRESHOLD` 结构体中没有这个成员。

根据当前工程结构，这里应改成实际存在的字段。若按当前阈值设计逻辑，建议改为：

```c
Threshold.SoliMin
```

修改前应再次确认该分支逻辑本来想控制的是哪一类阈值。

### 4. `SYS/menu/menu.c`

#### 问题：OLED 阈值页面显示项与实际调节项不一致

当前阈值控制模块的预期行为应为：

1. 第一项显示并调节温度阈值
2. 第二项显示并调节光照阈值
3. 第三项显示并调节土壤湿度阈值

也就是说，OLED 阈值页应统一为：

```c
Temp:
Light:
Soil:
```

并按以下字段顺序显示：

```c
Threshold.TempMax
Threshold.LightMin
Threshold.SoliMin
```

本地按键加减逻辑也必须和这个顺序完全一致，不能出现显示的是光照、实际调整的是土壤湿度，或者反过来的情况。

## 阈值控制模块的正确逻辑

对于当前自动浇花系统，OLED 页面中阈值控制模块应满足以下要求：

- 显示温度阈值
- 显示光照阈值
- 显示土壤湿度阈值
- 屏幕显示顺序与按键调节顺序一致
- 每次加减后，实际保存到 `Threshold` 结构体中的值也应同步变化
- 阈值不能越界，避免出现负数、下溢或异常大值

推荐采用如下映射：

- 温度：`Threshold.TempMax`
- 光照：`Threshold.LightMin`
- 土壤湿度：`Threshold.SoliMin`

## 当前结论

如果按本文档完成修改，则可以解决以下确定性问题：

- `memset` 调用错误导致缓冲区未清空
- 阈值加减操作无效
- OLED 浮点显示可能发生缓冲区溢出
- 串口阈值控制使用了不存在的结构体成员
- OLED 阈值显示顺序与实际调节顺序不一致

## 编译与检查建议

修改完成后，建议重新做以下检查：

1. 重新在 `Keil` 中编译工程，确认是否还有错误或警告
2. 重点检查 `mqtt.c`、`usart3.c`、`menu.c` 是否还有阈值相关警告
3. 运行后在 OLED 页面测试阈值调整
4. 检查温度、光照、土壤湿度三项是否能正确显示和增减
5. 检查串口或 MQTT 下发阈值后，OLED 与内部变量是否保持一致
